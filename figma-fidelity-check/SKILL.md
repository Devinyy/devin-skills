# Figma 还原度验证

对比 Figma 基准图和实现后的真实渲染图，并用可追溯的 Figma 数值定位差异。不得以肉眼估算间距或以未定义的百分比分数宣布达标。

## 输入

开始前确认：

1. Figma 链接：带 `node-id`，精确到目标 frame。
2. 实现来源：PC/H5 的可访问 URL，或小程序的开发者工具/真机截图。
3. 渲染配置：目标视口宽高、设备像素比、浏览器和字体加载方式。
4. 可选环境噪声基线：已验收页面的 Figma node、同一实现页面和相同渲染配置。没有基线时，报告必须标记“未建立”，不得豁免任何差异。

## 流程

### 1. 递归读取 Figma 约束

1. 使用 `get_screenshot` 截目标 node 作为基准图。
2. 先读取目标 frame 的 `get_design_context` 或等价结构元数据，再递归下钻所有 auto-layout 容器。
3. 对每个 auto-layout 容器读取 `layoutMode`、`gap/itemSpacing`、四向 `padding`、主/交叉轴对齐、尺寸模式和裁剪设置；对每个**直接子节点**单独读取设计上下文及尺寸/位置。
4. 遇到空白节点、spacer、绝对定位子节点或嵌套容器时，继续读取其约束，确认视觉空白来自何处。
5. 间距只引用父容器的 gap/padding 或节点的实际约束值；禁止根据截图写“约 16px”“看起来差不多”。无法取得的值标为“未读取到”，不要补估。

### 2. 校准渲染环境

1. 令浏览器视口、DPR、缩放、滚动条策略、字体加载和截图范围与 Figma frame 对齐；等待字体、数据和动画稳定后截图。
2. 有已验收基线时，按完全相同配置截 Figma 图和实现图，登记稳定出现的系统性差异，例如字体宽度或 hinting 的 1–2px 差异。
3. 后续报告只有在“属性类型、方向、量级和渲染配置”均与基线一致时，才将该差异标为“环境基线”；它不计为红黄项，但必须在报告中列出。
4. 没有已验收基线时，明确写“环境噪声基线：未建立”，不把任何差异归为环境噪声。

### 3. 获取实现渲染图

**PC / H5：**使用 Playwright 将视口设为目标尺寸，打开页面，等待稳定后截图。截图范围必须与 Figma frame 对齐。

**微信小程序原生：**让用户在开发者工具或真机按目标机型截图并提供机型/尺寸；若有小程序自动化或 H5 同构版本，可改走自动化截图。

### 4. 比对与判定

按布局/间距、尺寸、排版、颜色、圆角/描边、资源、状态和响应式逐项对比。每个差异必须包含“实现现状 → Figma 实际值 → 修改位置/建议 → 复验结果”。

使用以下唯一验收标准，不输出“整体 XX%”：

- 🔴 为 0。
- 🟡 最多 2 项，且每项都有修复建议并在后续复验中确认已执行。
- 所有间距类单项偏差不超过 2px。
- 颜色必须命中 Figma token；无 token 时必须精确匹配设计上下文的颜色值。

### 5. 输出与闭环

按 [报告模板](report-template.md) 输出。用户修改后，只复验上一轮红黄项及其相关父容器/直接子节点；直到满足枚举式验收标准。不得以截图相似度或主观总分取代该标准。

## 注意

- 一次只验证一个 frame/页面；多个页面分别建立报告。
- 报告只将可追溯的 Figma 数值作为“应为”；截图只用于确认视觉结果。
- 基线只用于排除已证明稳定的渲染噪声，不能覆盖真实的布局、资源、状态或 token 偏差。
