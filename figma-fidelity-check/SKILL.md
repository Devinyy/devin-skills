# Figma 还原度验证 (figma-fidelity-check)

把"实现后的真实渲染"和"Figma 设计稿基准图"两张图叠着比对,输出一份带严重程度和修复建议的还原度报告,并支持改完后再跑一轮形成闭环。

## 核心原则

还原度验证 = **两张图对比**,缺一不可:

```
Figma 设计稿截图(标准答案)   vs   实现后的真实渲染截图(实际结果)
```

只截 Figma 不算验证,只截实现也不算验证。必须两张都拿到再比对。

## 输入:开始前先确认

1. **Figma 来源**:Figma 链接(带 node-id 最好,精确到具体 frame/页面)。
2. **实现来源**:
   - PC / H5 / uni-app·Taro 编译出的 H5 → 可访问的 URL(本地 `localhost:xxxx` 或线上)。
   - 微信小程序原生 → 无法用浏览器打开,见下方「小程序模式」。
3. **目标尺寸/设备**:如 375×812(iPhone)、1440 宽(PC),用于对齐截图视口。

## 流程

### 第 1 步:取 Figma 基准图

- 用 figma MCP 的 `get_screenshot` 截目标 node(这是基准/标准答案)。
- 同时用 `get_design_context` 或 `get_variable_defs` 拿到精确规格:间距、字号、颜色、圆角、字重。
  纯肉眼比图会漏细节,**关键数值一定要从设计上下文里读**,而不是猜。

### 第 2 步:取实现渲染图

**PC / H5 模式(全自动):**
1. `browser_resize` 把视口设成目标尺寸(和 Figma frame 一致,如 375 宽)。
2. `browser_navigate` 打开实现页面 URL。
3. 必要时 `browser_wait_for` 等待数据/动画稳定。
4. `browser_take_screenshot`(整页或指定元素)。

**微信小程序原生模式(半自动):**
1. Playwright 打不开小程序(运行在微信开发者工具里,非标准浏览器)。
2. 让用户在微信开发者工具里:选好对应机型 → 截取当前页面 → 把图发进对话。
   - 提示用户尽量用「真机预览」或「模拟器」截图,并说明机型,以便和 Figma frame 尺寸对齐。
3. 拿到用户发来的截图后,与第 1 步基准图比对。
4. (可选进阶)若项目接了小程序自动化(miniprogram-automator)或有 H5 同构版本,可改走全自动模式。

### 第 3 步:比对并打分

逐项对照下面的清单,**按类别**列出差异,每条标严重程度(🔴严重 / 🟡中等 / 🟢轻微)并给出具体修复建议(含目标数值)。

比对清单:
- **布局/间距**:padding、margin、元素间距、栅格对齐、整体位置偏移。
- **尺寸**:容器宽高、组件大小、图标尺寸。
- **排版**:字号、字重、行高、字间距、文本截断/换行、对齐方式。
- **颜色**:文字色、背景色、边框色、渐变、透明度、阴影。
- **圆角/描边**:border-radius、border 宽度与颜色。
- **资源**:图片/图标是否正确、清晰度、是否变形、是否缺失。
- **状态**:hover/active/disabled/选中/空态/加载态(如设计稿有提供)。
- **响应式**:不同尺寸下的表现(如适用)。

### 第 4 步:输出报告

用 `report-template.md` 的结构产出报告:总体还原度评分 + 分类差异清单 + 优先修复项。

### 第 5 步:闭环

用户改完后,重新跑第 2~4 步,只对比上一轮标红/标黄的项,确认是否消除,直到达到验收标准(默认 🔴=0 且整体 ≥95%,可由用户调整)。

## 注意事项

- 截图视口尺寸必须和 Figma frame 尺寸对齐,否则像素级偏差全是噪声。
- 优先用设计上下文里的**真实数值**判断差异,而非纯目测。
- 一次聚焦一个 frame/页面,页面多时逐个过,不要混在一张报告里。
- 报告里每条差异都要可执行:说清"现状 → 应为 → 改哪里"。
