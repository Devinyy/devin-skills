# Frontend Functional Breakdown Workflow

## 目标

只根据 PRD，把业务需求转换为后续技术方案和估时可引用的前端功能地图。

## 文档结构

```md
# <需求名称> - 前端功能拆解

## 1. 需求范围
- 涉及端、各端本次新增 / 改造 / 复用的能力、明确不涉及项

## 2. 页面索引
| 端 | 页面 | 页面类型 | 变更类型 | 页面职责 |
|---|---|---|---|---|

## 3. <端名称>

### 3.1 <页面名称>
#### 模块：<模块名称>
| 字段 | 功能 | 逻辑 | PRD 来源 |
|---|---|---|---|

## 4. 跨页面逻辑

## 5. 待确认项
| 编号 | 所属端 / 页面 | 问题 | 影响 |
|---|---|---|---|
```

## 拆解规则

- 端：记录服务对象、覆盖业务范围，以及新增 / 改造 / 复用的能力
- 页面：记录名称、类型、职责和变更类型；不写路由或技术归属
- 模块：按业务区域拆分，不按组件或技术结构拆分
- 字段：仅记录 PRD 已明确的业务字段与规则
- 功能：记录用户可完成的业务动作，且必须回指 PRD
- 逻辑：记录 PRD 明确的业务判断、字段联动、状态流转、校验、计算与异常分支

## 边界

- 不写背景、目标、用户角色，除非直接影响功能范围或业务逻辑
- 不写 UI、控件样式、布局或交互细节；这些以 UI 设计稿为准
- 不写接口、数据模型、路由、权限接入、组件复用和状态管理
- PRD 未明确的内容统一进入待确认项，不得猜测或补全
