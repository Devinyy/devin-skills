# PRD Frontend Functional Breakdown

将 PRD 转换为一份可确认、可追溯的前端功能拆解文档。此 skill 在需求边界确认处停止；不输出技术方案、估时或飞书 Base 排期。

## 何时使用

- 用户给出 PRD 链接、PRD 原文或附件，要求梳理前端需求
- 用户需要按端、页面、模块、字段、功能、逻辑拆解 PRD
- 用户需要在技术方案或估时前确认前端功能范围与待确认项

以下场景不应使用：
- 用户只要阅读或概述 PRD，不需要前端功能拆解
- 用户已提供确认过的功能拆解，只需要技术方案、估时或 Base 回填
- 用户要的是后端详细架构设计

## 输入与输出

输入：PRD 正文、附件，或可读取正文的飞书 Wiki / Doc 链接。

默认输出：`docs/product/<topic>-frontend-functional-breakdown.md`

一份 PRD 默认只生成一份文档，文档内按端、页面组织；仅在超过 10–15 个页面或多人并行维护时，才按端拆分，并保留总览索引。

## 工作流

### 1. 获取 PRD 正文

优先级：
1. 用户直接提供 PRD 文本或附件
2. 用户提供飞书 Wiki / Doc 链接，先读取正文

只在成功获取正文后继续拆解，不要基于标题猜测内容。

### 2. 生成前端功能拆解

先阅读 [references/workflow.md](references/workflow.md)。

严格按以下层级梳理：

`端 → 页面 → 模块 → 字段 → 功能 → 逻辑`

- 先识别涉及的端；再按业务流程列页面；在页面内依次拆模块、字段、功能和逻辑
- 字段展示、功能与业务逻辑必须严格遵守 PRD
- 字段只记录 PRD 明确的业务字段及其展示、编辑、必填、默认值、格式、取值、校验与关联规则
- 逻辑只记录 PRD 明确的显隐、禁用、联动、状态流转、计算、操作前置条件与异常分支
- 跨页面业务规则集中写在“跨页面逻辑”中；页面内仅引用，避免重复维护
- PRD 未明确、相互冲突或无法落到上述层级的内容，统一列为待确认项，不得自行补全

### 3. 严格边界

此阶段只回答“前端需要做什么业务能力”，不得提前补充或推导：

- UI 视觉、控件形态、页面布局、动效或交互细节；这些必须以 UI 设计稿为准
- 路由、组件、状态管理、接口、数据模型、权限接入或其他技术实现；这些属于技术方案阶段
- PRD 中未明确的验收标准、业务规则或字段约束

## 输出标准

一份合格的前端功能拆解，至少要满足：

- 每项 PRD 内容都能追溯到端、页面、模块、字段、功能或逻辑
- 每个页面都有明确职责和变更类型（新增、改造或复用）
- 字段、功能和逻辑不超出 PRD 已明确范围
- 跨页面逻辑只维护一处
- 待确认项包含所属端 / 页面、问题和影响

## 常见坑

- 把背景、目标、用户角色作为正文；除非它们直接影响功能范围或业务逻辑
- 按技术组件而不是业务模块拆解
- 直接补接口、路由、权限或数据模型
- 根据没有设计稿的 PRD 猜测控件和交互细节
- 遇到缺失规则自行补全，而不是标为待确认项
