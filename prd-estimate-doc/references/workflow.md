# Workflow

## 标准产物

### 1. PRD 摘要
目标：把 PRD 压缩成后续可重复引用的本地 markdown。

建议结构：
- 背景与目标
- 用户角色
- 核心流程
- 功能范围
- 关键业务规则
- 风险与待确认项

### 2. 前端任务拆解
目标：把 PRD 转成前端实现清单。

建议结构：
- 按模块拆：基础模块、业务模块、系统设置、联调回归
- 每个模块下按页面/能力拆
- 如果后续要写回 Base，再继续拆成父任务 + 叶子任务

### 3. 前端技术方案
目标：解释任务拆解如何落到现有项目。

最少覆盖：
- 子应用或页面归属
- 路由位置
- 组件复用点
- 状态管理和上下文
- 接口组织和适配层
- 需要补的基础设施

### 4. 估时文档
目标：给出可解释的工时口径。

最少覆盖：
- 口径说明
- 分模块工时
- 开发 / 联调 / 回归拆分
- 风险冗余
- 关键假设

## 文件命名建议

按 topic 统一命名：
- `docs/product/<topic>-prd-summary.md`
- `docs/product/<topic>-frontend-task-breakdown.md`
- `docs/tech/<topic>-frontend-tech-design.md`
- `docs/project/<topic>-estimate.md`

`<topic>` 用稳定英文短名，避免后续重复生成多个近似文件。

## Base 落表建议

如果同步到飞书 Base：
- 明细表保留父任务和叶子任务两层
- 父任务工时留空
- 子任务写工时
- `阶段` 至少拆成：`开发`、`联调/自测`、`联调与全链路自测`
- 如果排 AI 协同工期，再加：`AI直接实现`、`执行方式`、`执行波次`
