# PRD Frontend Tech Design

编写本次需求的前端落地方案，不写脱离仓库与需求事实的通用架构文章。只写评审时真正会被追问的内容：页面与路由、核心流程、组件复用、关键状态与数据流中的前端职责、约束与兜底、核心接口清单、单测点。不复述 PRD，不重写团队既有约定。

## 输入与准备

按优先级获取并交叉核对：

1. PRD 原文：背景、目标、用户场景、业务规则、验收条件。
2. 前端功能拆解：功能范围、页面和任务边界。
3. `docs/product/<需求名>-figma-info.md`：页面状态、组件、资源和设计覆盖缺口。
4. 目标仓库：模块边界、路由、组件库、状态/请求层、现有接口约定、测试和发布方式。
5. API 文档或接口清单：请求/响应、鉴权、异常和幂等要求。

先检查输入冲突和缺口。对会改变方案边界的事项（目标子应用或仓库、关键用户路径、接口归属/契约、权限模型、上线策略）先向用户确认；其余未知信息记录为待确认，禁止编造接口字段、性能指标、排期、仓库路径或兼容范围。

## 编写流程

1. 从 PRD 提炼两三句概述与改动范围，链接 PRD 原文；不复述业务目标和用户场景。
2. 将功能拆解与 Figma 信息映射到页面、路由和组件。设计稿未覆盖的功能或 PRD 未定义的交互，必须在对应章节就地标注为待确认。
3. 阅读目标仓库，确定子应用/模块、路由、状态与店铺/租户/权限上下文、请求适配层和需要新增的基础设施。逐页面/功能模块形成组件复用结论，按下方规则记录。
4. 描述核心流程。涉及 3 个以上步骤、分支或异步状态时用 Mermaid；简单流程用文字，避免为了画图而画图。默认“页面与路由”在“核心流程图”之前（先建立“改哪里”的骨架，再看“怎么跑”）；需求以单页面内复杂交互为主、跨页面跳转很少时，可将流程图前置。
5. 写清关键业务状态与数据流中的前端职责边界：哪些校验在前端做、哪些异常由前端兜、状态由谁驱动、前端持有和派生哪些状态。
6. 汇总约束、兜底与注意点：接口未定、设计缺口、兼容性、并发/时序、历史数据等风险统一写在这里，每条写明兜底或规避策略，不单独设风险评估章节。
7. 梳理核心接口清单。未确认接口标“待确认契约”，不写虚构 JSON 示例。
8. 列出单测点，落到具体可测的行为和边界，不写“功能/回归/兼容/性能”式的空表。
9. 按 [输出结构](references/output-schema.md) 生成文件，并做一致性检查：技术方案、功能拆解、Figma 信息和接口清单之间不应出现互相矛盾的页面、状态或功能范围。

## 输出规则

- 文件必须命名为 `docs/tech/<需求名>-frontend-tech-design.md`；`<需求名>`保留用户给出的名称。
- 使用中文 Markdown。链接 PRD、Figma 信息、代码仓库文件和接口文档时保留原始来源。
- 每项结论标清事实来源；将“现状”“本次新增/修改”“待确认”区分开。
- 待确认项采用双层机制：在出现处就地标注（如“⚠️ 待确认：…”），结尾“待确认问题总表”只做汇总索引并注明出现章节，不做唯一记录点。禁止把 gap 只藏在结尾总表里。
- 组件库默认使用仓库现有组件库；只有在已确认的项目语境中才可写 `dcgj-ui`。
- “组件复用方案”不得写“复用现有组件”“参考某模块实现”等笼统结论。每个页面/功能模块必须逐项写明：现有组件的精确文件路径、基础组件库组件名称、结论（`直接复用` / `不复用` / `新增业务组件`）、判断依据与使用边界。
- 只有确认现有组件的 props、事件、状态与数据契约可覆盖需求时，才可标为“直接复用”；例如单选能力不能用于多选筛选，固定邮箱导出弹窗不能用于导出方式尚未定义的需求。
- 标记“不复用”时，说明字段语义、交互能力、数据契约、业务耦合或固定样式中的具体不满足项；标记“新增业务组件”时，写明组件名称、归属目录与责任边界。不得为复用修改全局组件，也不得跨子应用引用组件。
- PC 端优先使用项目 `dcgj-ui` 的二开组件；小程序端使用项目已有 `uview-plus` 和本端组件体系，禁止混用。仓库不可访问或无法确认路径/能力时，直接提问，不得虚构路径、组件名或复用结论。
- 技术方案说明如何实现与验证，不承担估时或 Base 排期；需要估时应交给独立估时技能。
- 不写团队既有约定（分支规范、协作流程、发布流程）；灰度/监控/回滚仅在需求明确要求时，以注意点形式写入“约束、兜底与注意点”。

## 完成标准

另一位前端工程师应能据此定位改动位置、理解关键流程与前端职责边界、识别设计和需求缺口、完成联调与验收；评审者应能在每个章节就地看到该处的待确认项，并在结尾总表逐条销项。
