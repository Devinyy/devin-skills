# PRD Frontend Tech Design

编写本次需求的前端落地方案，不写脱离仓库与需求事实的通用架构文章。输出结构基于团队“需求技术方案设计文档”模板。

## 输入与准备

按优先级获取并交叉核对：

1. PRD 原文：背景、目标、用户场景、业务规则、验收条件。
2. 前端功能拆解：功能范围、页面和任务边界。
3. `docs/product/<需求名>-figma-info.md`：页面状态、组件、资源和设计覆盖缺口。
4. 目标仓库：模块边界、路由、组件库、状态/请求层、现有接口约定、测试和发布方式。
5. API 文档或接口清单：请求/响应、鉴权、异常和幂等要求。

先检查输入冲突和缺口。对会改变方案边界的事项（目标子应用或仓库、关键用户路径、接口归属/契约、权限模型、上线策略）先向用户确认；其余未知信息记录为待确认，禁止编造接口字段、性能指标、排期、仓库路径或兼容范围。

## 编写流程

1. 从 PRD 写出业务目标、技术目标和“谁 → 做什么 → 达成什么”的用户场景。SMART 指标仅在原始输入提供时保留，不自行量化。
2. 将功能拆解与 Figma 信息映射到页面、路由、组件、状态和数据流；设计稿未覆盖的功能或 PRD 未定义的交互必须显式标记。
3. 阅读目标仓库，确定子应用/模块、路由、复用组件、状态与店铺/租户/权限上下文、请求适配层、错误处理与需要新增的基础设施。
4. 描述关键业务和系统流程。涉及 3 个以上步骤、分支或异步状态时用 Mermaid；简单流程用文字，避免为了画图而画图。
5. 对每个核心功能说明组件职责、数据来源和流向、交互与异常状态、接口依赖以及版本/兼容处理。必要时给出接口表；未确认接口用“待确认契约”，不写虚构 JSON 示例。
6. 列出风险、质量保障、灰度/监控/回滚、开发协作和验收。仅在需求或仓库支持时写具体方案，否则说明不适用或待确认。
7. 按 [输出结构](references/output-schema.md) 生成文件，并做一致性检查：技术方案、功能拆解、Figma 信息和接口表之间不应出现互相矛盾的页面、状态或功能范围。

## 输出规则

- 文件必须命名为 `docs/tech/<需求名>-frontend-tech-design.md`；`<需求名>`保留用户给出的名称。
- 使用中文 Markdown。链接 PRD、Figma 信息、代码仓库文件和接口文档时保留原始来源。
- 每项结论标清事实来源；将“现状”“本次新增/修改”“待确认”区分开。
- 组件库默认使用仓库现有组件库；只有在已确认的项目语境中才可写 `dcgj-ui`。
- 技术方案说明如何实现与验证，不承担估时或 Base 排期；需要估时应交给独立估时技能。
- 不适用的可选章节（分层架构、性能专项、兼容、灰度、监控、回滚）写“不适用 + 原因”，不要套用示例措辞。

## 完成标准

另一位前端工程师应能据此定位改动位置、理解关键流程与数据契约、识别设计和需求缺口、完成联调与验收；评审者应能追溯每个关键决策的输入依据和风险处置。
