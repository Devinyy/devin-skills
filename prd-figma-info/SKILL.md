# PRD Figma Info

从 Figma 采集可追溯的设计事实，输出给前端功能拆解和技术方案使用；不要把设计稿截图或主观猜测当作元数据。

## 输入与边界

需要以下输入：

- 需求名称：用于文件名；保留用户给出的名称，不自行翻译或缩写。
- Figma 文件或节点链接：至少能定位文件；优先使用包含 `node-id` 的链接。
- PRD 或前端功能拆解：用于核对设计覆盖，而不是从中补造设计数据。

先确认采集范围。链接未指定页面、画板或关键流程且文件含多个无关需求时，先列出候选范围并请用户指定；不要全量采集整个设计文件。只有一个明显关联页面时，可以将其作为默认范围，并在结果中说明。

## 采集流程

1. 记录设计来源：文件 URL、文件/页面/节点名称、节点 ID、采集时间和采集范围。
2. 读取目标页面或画板的结构元数据，保留可追溯的层级、节点名称、类型、尺寸和布局关系。调用 Figma 工具前，先加载 `figma:figma-use`；该技能规定 `use_figma` 的用法。优先使用结构/元数据读取能力，截图仅用来辅助核对视觉或交互。
3. 按用户路径归纳页面和状态：入口、页面/弹窗/抽屉、加载/空/错误/禁用/权限不足等状态，以及可见的跳转、关闭、提交和返回行为。
4. 提取实现相关的设计数据：布局与断点、栅格/间距、颜色与字号、组件/实例/变体、图标/图片/导出资源、表单和表格规则、可访问性提示、原型连线或设计标注。
5. 将设计事实压缩为前端可执行说明：复用组件、页面状态、数据展示和交互约束。事实与推断分开写；对接口字段、权限逻辑或业务规则等设计稿无法证实的内容标记为“待确认”。
6. 将 PRD 或功能拆解逐项与设计范围比对，标记“已覆盖”“部分覆盖”“未见设计稿”“仅设计稿出现”。不要因未见设计稿而推断功能不存在。
7. 按 [输出结构](references/output-schema.md) 生成文件。完成后核对链接/节点、关键页面状态、组件与资源、覆盖结论均可回溯。

## 输出规则

- 文件必须命名为 `docs/product/<需求名>-figma-info.md`；`<需求名>`与用户提供的名称一致。
- 使用中文 Markdown，标题明确，保留原始 Figma 链接与节点标识。
- 对大量重复节点，汇总为组件或模式；不要粘贴完整原始节点树、序列化 JSON 或截图二进制。
- 将“设计事实”“实现建议”“待确认”分段表述。实现建议只说明设计对前端的约束，不替代技术方案。
- 读取受限、链接失效、缺少目标范围、不可导出资源或原型不可访问时，保留已获取内容，并在“采集限制与待确认”中说明。

## 完成标准

结果应使另一位前端工程师无需重新翻找设计稿，也能定位设计来源、理解页面与状态、复用组件与资源，并识别 PRD/设计稿之间的缺口。
