# 前端估时

## 输入与口径

先读取技术方案、PRD 或已完成的功能拆解；技术方案与 PRD 冲突时，以用户指定的版本为准。没有明确范围、目标端、页面归属、支付接入点、接口契约或设计稿时，不得编造：标为“待确认”，并将会改变工作量的事项先向用户提问。

默认采用真人开发口径：包含前端开发、联调、自测、全链路回归和发布准备；不包含后端开发、产品澄清等待、设计稿返工和未确认的新增范围。交付时明确总工时、各端工时和排除项。

## 拆解规则

使用严格层级：**端 → 页面 → 模块**。

- **端**：使用表内真实客户端字段，例如运营端、小程序、商家端、H5；不能把仓库名当作端。
- **页面**：每个有独立路由、入口或详情容器的页面建立一个父任务。父任务不填工时。
- **模块**：叶子任务只描述一个页面内可独立开发、验证的能力，例如筛选查询、表格操作、表单分区、详情信息、弹层操作、资产流水、状态处理。
- **UI 开发**：每个新增或改造的业务页面都必须单独创建至少一条 UI 开发叶子任务，不能将其隐含在功能开发中。UI 任务应覆盖页面布局、项目组件库适配、正常/加载/空/错误状态和必要的端内适配；按页面复杂度拆分为多个 UI 模块。
- **跨页面能力**：路由/API/类型、组织上下文、联调、自测、全链路回归和发布等不强行归属业务页面；建立端级“公共基础”或“联调与自测”父任务，并在任务名称说明影响范围。
- **复用**：根据技术方案写明实际复用的项目组件/页面/API，而不是写“复用现有组件”的泛化描述。

不要按接口、文件、按钮或视觉区块机械拆分；以能独立完成并验收的模块为最小粒度。一个叶子任务只归属一个端、一个页面父任务和一个阶段。

未提供 Figma 时，仍必须保留 UI 开发任务，任务名称追加“设计稿待确认”。此时只估算复用现有组件库和页面约定完成基础布局/状态的工作量；不得虚构尺寸、样式或交互，也不得把后续 Figma 还原、视觉验收和返工纳入该基础 UI 工时。拿到设计稿后，补充或调整对应 UI 任务并重新汇总。

## 工时硬约束

- 阶段为“开发”的叶子任务必须 **小于 4h**；`4h` 不合格，必须继续拆分为两个或以上子任务。
- “联调/自测”与“联调与全链路自测”阶段可为 `>= 4h`，但应按端或主流程拆开，避免笼统的“全量测试”。
- 父任务、模块容器和汇总记录的工时必须留空，避免重复累计。
- 开发、接口联调、自测回归、全链路回归/发布准备必须分别建任务，不能把它们吸收到页面开发工时。
- UI 开发属于“开发”阶段，同样必须 `< 4h`；复杂表单、长列表或多状态页面应拆为页面布局、组件/状态呈现、弹层/适配等独立 UI 任务。
- 工时使用 0.5h 为最小粒度；任何包含多个独立页面或核心流程的任务均需拆分。

## 飞书 Base 工作流

用户给出 Base/Wiki 链接时，使用 `lark-base`：

1. 用 `base +url-resolve` 获取真实 `base_token`、`table_id`、`view_id`；不得把 Wiki token 当作 Base token。
2. 用 `table-get`、`field-list` 和少量 `record-list` 读取真实表结构、选项值、父子关联字段与模板粒度。
3. 用户说“以模板为准”时，优先 `base +base-copy --without-content` 新建需求专属估时表，不修改原模板；除非用户明确要求写入原表。
4. 先创建页面父任务，再根据返回的 `record_id` 批量创建叶子任务并写入父记录关联。只写可写字段；选择项必须使用表中真实选项。
5. 写入后用 `data-query` 在云端按端、阶段汇总工时，确认父任务无工时、叶子任务均关联父记录，并报告结果链接。

常见模板字段映射：`客户端`=端，`主模块`=页面，`任务`=页面内模块，`父记录`=页面父任务，`工时(h)`=叶子任务工时，`阶段`=开发/联调自测/全链路自测。若实际字段不同，以读取到的字段为准，不自行新建或假设字段。

## 输出与验收

输出或写入前逐项检查：

- 每条开发叶子任务 `< 4h`；联调、自测例外已明确标注阶段。
- 每个业务任务都能映射为“端 → 页面 → 模块”，跨页面任务已归入公共基础或联调模块。
- 每个新增或改造的业务页面均有独立 UI 开发叶子任务；单独汇总 UI 工时，并声明设计稿是否已提供。
- 父任务无工时，叶子任务有且仅有一个父记录。
- 接口、设计稿、权限、支付等未确认内容标为待确认，并说明是否未纳入估时。
- 报告总工时、按端汇总、按阶段汇总、任务数、主要风险与待确认项。
