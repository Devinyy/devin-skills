# Figma Tech Design to Code

将 Figma 设计和前端技术方案落为目标仓库中的代码。技术方案定义业务、模块、路由、数据和边界；Figma 设计上下文定义视觉与交互事实；现有项目决定组件、token 和工程写法。三者冲突时先停下确认，不擅自选择。

## 何时调用

在以下场景调用：

- 用户提供 Figma 节点链接和前端技术方案，要求在指定项目中实现页面、模块或交互。
- 用户说“按技术方案还原 Figma”“将这个设计稿落成代码”“基于 Figma 开发并接入现有项目”。
- 用户要求从设计稿实施代码，并要求复用项目组件、组件库或完成还原度验收。

不用于仅提炼 Figma 页面/组件/交互摘要、仅编写技术方案，或只做已完成页面的视觉验收；这些分别使用 `prd-figma-info`、`prd-frontend-tech-design` 和 `figma-fidelity-check`。

## 疑问先提问

任何会影响代码范围、业务行为、组件选择或验收结论的不确定项，都直接向用户提出一个清晰的问题后再继续；不得自行补全。包括但不限于：目标 Figma 节点、技术方案对应范围、目标仓库/子应用、路由入口、接口契约、权限与上下文、目标状态、组件库选择、资源归属、运行验证方式和基线样本。

## 必要输入

- 带 `node-id` 的 Figma 链接；文件级链接不足以实施，必须要求用户补充目标节点。
- 前端技术方案：优先 `docs/tech/<需求名>-frontend-tech-design.md`。
- 目标仓库和可运行/验证方式。

可选输入：`<需求名>-figma-info.md`、接口文档、已验收页面的 Figma node 与实现地址（环境噪声基线）。

## 实施流程

### 1. 先对齐方案与项目

1. 阅读技术方案、Figma 信息和接口资料，列出本次需要实现的页面、路由、状态、接口依赖与验收条件。
2. 检查目标仓库的模块边界、路由、状态/请求层、样式体系、设计 token、测试方式和已有同类页面。
3. 将技术方案映射到实际代码位置。方案未明确仓库位置、接口契约、权限或关键状态时，先向用户确认；不得以 Figma 外观补造业务逻辑。

### 2. 复用优先

编码前按以下优先级检索并记录复用结论：

1. Figma Code Connect 映射的项目组件。
2. 项目内语义、交互和视觉均匹配的已有组件或同类页面模式。
3. 项目已接入的组件库及其 token、主题和标准交互。
4. 仅在前三项均不满足时新建本地组件，并说明不可复用原因。

不得因名称相似就复用图标或组件；确认其状态、尺寸、可访问性和交互均匹配。不得绕过组件库手写已有的按钮、表单、弹窗、表格或图标。图标、图表优先检索并复用项目内或组件库已有能力；仅在不存在且需求/设计明确需要时才新增。禁止为装饰、状态或操作入口随意添加 Emoji 表情；只有 Figma 或产品文案明确要求时才可保留。

**切图上传规则：**Figma 节点名匹配 `icon/*`、`icon_*`，或节点带 slice/export 切图标记时，将其视为指定切图资产，不用名称相近的组件库图标替代。必须自动调用 `qiniu-icon-upload`：从 `get_design_context` 的资源 URL 导出透明 PNG（不得使用截图）、按页面或公共资源分类上传七牛，并使用上传结果的 CDN `url` 替换代码引用。不得保留本地 `/static/icon` 路径、临时 Figma 资产 URL 或自行绘制的替代图标。七牛配置缺失、资源格式不支持或页面分类不明确时，直接向用户提问后再继续。

### 3. 递归读取 Figma，禁止目测

1. 调用 `get_design_context` 前，必须先加载 `figma:figma-design-to-code`，并在调用的 `skillNames` 中包含该技能名。
2. 对目标 node 调用 `get_design_context` 后，递归下钻所有 auto-layout 容器；只读取顶层 frame 即停止属于失败。
3. 每个 auto-layout 容器都读取 `layoutMode`、`gap/itemSpacing`、四向 `padding`、主/交叉轴对齐、尺寸模式和裁剪设置；每个直接子节点都单独调用 `get_design_context` 或等价设计上下文读取。
4. 遇到 spacer、空白节点、绝对定位节点或嵌套容器时继续读取，确认视觉空白与定位来源。
5. 间距、尺寸、字体、行高、颜色、圆角和阴影只使用设计上下文的实际值；禁止依据截图写“约”“看起来差不多”或自行估算。读取失败时停止并说明缺失值，不以目测补齐。

### 4. 编码

1. 将 Figma 返回的参考代码转换为目标项目的框架、语言、组件库、样式和 token；不得直接粘贴 React + Tailwind 参考代码。
2. 按技术方案实现路由、权限/店铺/租户上下文、数据请求、加载/空/错误/禁用状态、交互和异常处理。
3. 保留 Figma 中可见的状态与交互；对设计稿未覆盖但技术方案要求的状态，按项目既有模式实现并标注其依据。
4. 修改后运行项目已有的格式化、类型检查、测试和构建命令；按项目约定补充或更新测试。

### 5. 校准并验收

1. 调用 `figma-fidelity-check` 做还原度验证，且令视口、DPR、缩放、滚动条策略、字体加载和截图范围与 Figma frame 对齐。
2. 有已验收样本时，先建立环境噪声基线；只有属性类型、方向、量级和渲染配置均与基线一致的稳定差异，才可标为环境噪声并从红黄项排除。
3. 没有已验收样本时，报告必须写“环境噪声基线：未建立”，不得豁免任何差异。
4. 仅在以下条件全部满足时宣布达标：🔴=0；🟡≤2 且均已修复并复验；每项间距偏差≤2px；颜色命中 Figma token（无 token 时精确匹配设计色值）。禁止使用未定义的“整体 ≥95%”或主观相似度。

## 交付内容

- 目标仓库中的实现代码、资源和必要测试。
- 复用记录：项目组件、组件库组件或新建组件的选择与原因。
- 验证结果：执行过的检查、Figma 约束取数、还原度报告、基线状态及未解决项。

## 失败与待确认

- 缺少 `node-id`、Figma 上下文读取失败或未读取到关键约束时，不要从截图猜实现。
- 技术方案与 Figma 在页面范围、状态、接口或权限上冲突时，输出差异并请求确认。
- 组件库或项目组件能力不足时，先说明缺口和新建组件方案，再继续实现。
